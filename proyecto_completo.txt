2. Resumen Ejecutivo

2.1. Objetivo del proyecto de software
Desarrollar una plataforma web que permite crear formularios personalizables, recopilar respuestas de forma centralizada y generar visualizaciones simples en tiempo real, como proyecto de aprendizaje universitario aplicado a problemas reales.
2.2 Alcance del sistema desarrollado
El alcance del sistema Form-Creator incluye la creación de una aplicación web que cubre todo el ciclo de vida de un formulario, desde su creación hasta la visualización de respuestas mediante gráficos dinámicos o su cierre definitivo. La plataforma permite la gestión para dos roles principales: usuario administrador (con todas las funcionalidades de creación, edición, publicación, duplicación, eliminación y análisis de formularios) y respondedor (responder formularios vía enlace, editar respuestas si está configurado y recibir copias por correo). No abarca apps móviles nativas, análisis estadístico avanzado, integraciones con CRM/ERP, pagos o colaboración en tiempo real.

2.3. Principales prácticas de gestión aplicadas
Form-Creator ha sido concebido bajo una metodología Híbrida que integra elementos de Scrum (Ágil) con prácticas de Gestión Tradicional de Proyectos, específicamente adaptada para los requisitos del proyecto.
2.4. Resultados obtenidos
Se completaron cuatro sprints que permitieron implementar progresivamente autenticación de usuarios, diligenciamiento de formularios públicos, envío de copias por correo y un panel de análisis con gráficos. El sistema final incluye backend en Django con MongoDB y frontend en React, proporcionando funcionalidades completas de gestión de formularios, recolección de respuestas y visualización de datos.
3. Introducción

Contexto del sistema desarrollado
Es una plataforma web responsiva con frontend en React (para interfaces modulares y gráficos con Recharts), backend en Django 5.1.2 con Django REST Framework (para APIs RESTful y lógica de negocio), y base de datos MongoDB (para datos flexibles de formularios y respuestas). Soporta roles como administrador (crea/gestiona) y respondedor (responde/edita si se permite), con autenticación JWT/Google, generación de enlaces, validaciones y exportación CSV. Desplegada en Vercel (frontend), Railway (backend) y MongoDB Atlas.
Importancia de aplicar procesos de ingeniería de software
Se aplican procesos para garantizar estructura, trazabilidad y calidad: desde acta de constitución y SRS, hasta modelado BPMN, casos de uso, backlog priorizado y sprints iterativos. La metodología híbrida justifica formalidad inicial (planificación) con flexibilidad ágil (ejecución en 4 sprints), reduciendo riesgos, alineando requisitos y permitiendo mejoras continuas vía reviews y retrospectivas. Esto asegura mantenibilidad, escalabilidad y cumplimiento de objetivos universitarios aplicados a problemas reales.
Temas abordados
Cubre análisis (visión negocio, requisitos funcionales/no funcionales, casos de uso, épicas/historias de usuario), diseño (arquitectura 3 capas, modelo de datos, UI/UX con mockups, APIs), desarrollo (cronograma, sprints, tecnologías), resultados, conclusiones y anexos (manuales, código en GitHub, video demo). Incluye diagramas (BPMN, secuencia, clases), matrices de trazabilidad y 15 historias de usuario priorizadas.

4. Descripción General del Proyecto de Software

4.1. Descripción del sistema
Nombre del Sistema: Plataforma de Encuestas y Formularios para Empresas (Form Creator).
Propósito Principal: El producto de software desarrollado ofrece una serie de funcionalidades destinadas a optimizar la gestión de formularios y la gestión de sus respuestas. 
Usuarios del sistema: Administradores, usuarios.
4.2. Alcance del proyecto
El alcance del proyecto incluye la creación de una aplicación web que cubra todo el ciclo de vida de un formulario, desde su creación hasta la visualización de sus respuestas por gráficos dinámicos, o bien por defecto, su cierre. La plataforma permitirá la gestión de un usuario administrador y un respondedor, donde el administrador tendrá habilitado todas las funcionalidades de gestión de un formulario y el respondedor tendrá habilitado responder un formulario, editar su respuestas enviadas (si el administrador planteó esa configuración)  y recibir una copia de sus propias respuestas al correo. 

4.3. Objetivos del proyecto
Objetivo General
Desarrollar una plataforma web que permite crear formularios personalizables, recopilar respuestas de forma centralizada y generar visualizaciones simples en tiempo real, como proyecto de aprendizaje universitario aplicado a problemas reales.
Objetivos Específicos
Desarrollar un constructor de formularios dinámicos que facilite la creación de diversos tipos de campos y estilos.
Implementar un sistema de generación de enlaces únicos para cada formulario, incluyendo la gestión de ventanas de recolección de datos, y validar su funcionamiento con pruebas automatizadas.
Crear un dashboard de resultados que presenta gráficos interactivos (barras, líneas y pastel) con opciones de filtrado por tiempo y categorías.

6. Gestión de Cambios del Software
6.1 Proceso de solicitud de cambios
Todo cambio al sistema Form-Creator debe iniciarse mediante el diligenciamiento de un Formulario de Solicitud de Cambio (CRF), identificado con un número consecutivo (por ejemplo, Solicitud de Cambio N.° 01 – Exportación a PDF de gráficos). El proceso es gestionado por el Comité de Control de Cambios (CCB), conformado por el Gestor de Configuración (Juan David Mendoza Ropero) y el Líder de Proyecto (Nicolás Felipe Meneses Peinado).
El CRF debe contener, como mínimo:
Nombre del proyecto y número de solicitud
Datos del solicitante: nombre, cargo, área y fecha
Descripción detallada del cambio solicitado
Justificación del cambio
Análisis de impacto preliminar: alcance, cronograma, costo, calidad y otros riesgos (recursos, adquisiciones, etc.)
Una vez registrada, la solicitud pasa al estado "En análisis" y es revisada por el CCB, quien la aprobará o rechazará con base en los criterios definidos. Los cambios aprobados son planificados en Jira y avanzan por los estados: Propuesto → En análisis → Aprobado/Rechazado → En implementación → En pruebas → Cerrado.

6.3 Evaluación de impacto
Antes de aprobar cualquier cambio, el Gestor de Configuración y el equipo técnico analizan su impacto en los siguientes ejes:
Impacto en tiempo:
Cada cambio aprobado requiere tiempo adicional de análisis, implementación, pruebas y actualización de documentación. El CCB evalúa si el cambio cabe en el sprint actual o debe planificarse en una iteración posterior, asignando responsables y fechas en Jira.
Impacto en costo:
Se evalúa si el cambio implica la incorporación de nuevos servicios de pago, licencias o recursos computacionales adicionales. Para Form-Creator, la política es mantener el uso de servicios gratuitos (Vercel, MongoDB Atlas, Railway en nivel gratuito); cualquier cambio que supere este límite deberá justificarse explícitamente ante el Líder de Proyecto.
Impacto en funcionalidad:
Se identifican los ítems de configuración (CI) afectados en el CMR: módulos de frontend (CI-FE-01), backend (CI-BE-01), base de datos (CI-DB-01), autenticación (CI-AU-01), notificaciones (CI-EM-01) y modelos de dominio. Se verifica que el cambio no rompa funcionalidades ya implementadas mediante análisis de dependencias y pruebas de regresión.
Impacto en arquitectura:
Se revisa si el cambio modifica la arquitectura actual (React 19 + Django 5.2 + MongoDB + Firebase Authentication + Brevo SMTP), el modelo de datos, la especificación OpenAPI (CI-API-01) o el design system (CI-UI-01). Los cambios que afecten CIs de nivel Crítico (backend, base de datos, autenticación) requieren aprobación explícita del CCB antes de iniciar cualquier implementación.

6.4 Control y aprobación de cambios
Procedimiento de aprobación
El CCB revisa cada solicitud en reunión ordinaria (frecuencia semanal en fases de alto desarrollo, quincenal en mantenimiento) o en reunión extraordinaria para cambios urgentes de alto impacto.
Los criterios de decisión del CCB son:
Alineación estratégica: el cambio debe estar alineado con los objetivos de negocio de Form-Creator (simplificar recolección de datos, democratizar análisis, eliminar barreras técnicas).
Impacto en alcance, cronograma, costo y calidad: se evalúa el efecto sobre el plan vigente.
Impacto en CIs críticos y experiencia de usuario: cambios sobre CI-BE-01, CI-DB-01 o CI-AU-01 reciben escrutinio especial.
Riesgos técnicos y operativos: se identifican supuestos de riesgo (p. ej. mal formato de PDF, degradación de rendimiento por conversión de gráficas).
Disponibilidad de recursos: se verifica la capacidad del equipo en el sprint o iteración actual.
Dependencias con otros cambios: se revisan conflictos o dependencias con otras historias de usuario en el backlog.
Solo los cambios que superen satisfactoriamente todos estos criterios serán aprobados para implementación. La decisión (Aprobado/Rechazado) queda registrada en el ticket Jira correspondiente con fecha, concepto de aprobación y firma del Líder de Proyecto.
Criterios de aceptación para cierre del cambio
Un cambio sólo puede cerrarse como "Cerrado" cuando se cumplan todos los siguientes criterios: 
Todas las tareas y subtareas en Jira completadas, sin incidencias abiertas relacionadas
Pruebas unitarias, de integración y de regresión exitosas, con evidencias registradas (capturas, reportes)
Documentos actualizados: SRS (EP2), ADD (EP4), Modelo de datos (EP6), Manual del sistema y Manual de usuario
CIs actualizados en el CMR con nueva versión y estado Liberado
Auditoría de configuración sin no conformidades críticas relacionadas con el cambio
Aprobación explícita del Gestor de Configuración y del Líder de Proyecto

7.2 Control de versiones
El control de versiones se realiza utilizando Git con un repositorio alojado en GitHub en la URL: https://github.com/nicolasmmm7/form-creator.
Este repositorio almacena el código fuente del backend y frontend, permitiendo llevar un historial detallado de cambios mediante commits, comparación entre versiones y reversión en caso de introducir errores.
Identificación de versiones: cada commit representa una versión de los archivos modificados, con identificador único, autor, fecha y mensaje que explica el cambio realizado.
Gestión del repositorio de versiones: Git centraliza las versiones aprobadas del código, evitando el “problema de la copia correcta” al asegurar un único repositorio autorizado.
Historial y trazabilidad de cambios: se puede consultar qué se cambió, quién lo hizo, cuándo y en qué archivos, lo que facilita auditorías de configuración y análisis de impacto.
Soporte al despliegue: la integración con plataformas como Vercel y Railway se basa en ramas del repositorio, lo que permite relacionar un despliegue concreto con una versión exacta del código.
7.3 Estrategia de ramas
Se propone para Form‑Creator una estrategia de ramas sencilla, adecuada al tamaño del equipo y al contexto académico, que favorece la estabilidad de la rama principal y la experimentación controlada.
Ramas principales:
main (rama principal/producción): Contiene siempre una versión estable del sistema, asociada a las entregas formales (cierre de sprints, demos y versiones finales presentadas en Análisis y Diseño e Ingeniería de Software). Solo se integra en main código que ha sido probado y validado por el equipo, reduciendo el riesgo de romper la versión utilizada para despliegues en Vercel y Railway.

Ramas de desarrollo por funcionalidad (feature branches): Para nuevas historias de usuario o módulos (por ejemplo, autenticación, creación de formularios, responder formularios, estadísticas), se crean ramas temporales del tipo feature/hu-XXX-descripcion. En estas ramas se realiza el desarrollo y pruebas iniciales; una vez completado y revisado el cambio, la rama se integra en main mediante merge o pull request, asegurando revisión previa.

Ramas de corrección rápida (hotfix, opcional): Ante errores críticos detectados en una versión ya desplegada, se puede crear una rama hotfix/descripcion a partir de main para corregir el problema de forma aislada y luego fusionarla de nuevo en la rama principal.

Esta estrategia de ramas permite:
Trabajar en paralelo en diferentes funcionalidades sin afectar la estabilidad de main.
Mantener trazabilidad entre historias de usuario/casos de uso y las ramas donde se implementan.
Facilitar auditorías y revisiones técnicas, al poder aislar el conjunto de cambios asociados a una funcionalidad o corrección concreta.

7.4 Baselines del proyecto
En el proyecto Form‑Creator se reconocen las siguientes líneas base relevantes, alineadas con los hitos académicos y de desarrollo:
Línea base 1 – Documentación de Análisis y Diseño (ADIS)
Momento: cierre de la materia de Análisis y Diseño de Sistemas.
Contenido principal: documento final de Análisis y Diseño de Form‑Creator, especificación de requisitos, casos de uso, modelos BPMN, modelo de dominio, modelo lógico de datos y diseño de arquitectura de alto nivel.
Propósito: servir como referencia estable para el inicio de la implementación y para la trazabilidad entre requisitos y código desarrollado.
Línea base 2 – Versión funcional intermedia (cierre de sprints clave)
Momento: cierre de los sprints en los que se completaron autenticación, creación de formularios y respuesta de formularios (por ejemplo, tras sprint 2–3 descritos en el resumen ejecutivo).
Contenido principal:
Código backend y frontend en un estado estable en la rama main, desplegado en ambientes de prueba/producción (Railway y Vercel).
Esquema de base de datos consolidado para gestionar usuarios, formularios y respuestas.
Documentación actualizada de historias de usuario, casos de uso y validaciones implementadas.
Propósito: establecer un punto de referencia al que se pueda volver si una funcionalidad avanzada introduce problemas, manteniendo una versión demostrable y coherente del sistema.
Línea base 3 – Versión final de producto (cierre de Ingeniería de Software)
Momento: entrega final del proyecto de Ingeniería de Software, con el sistema completo y documentación de gestión, calidad y configuración.
Contenido principal:
Código fuente del sistema con todas las funcionalidades comprometidas (gestión completa de formularios, envío de correos, panel de análisis con gráficos y filtros dinámicos), en estado estable.
Artefactos de despliegue definitivos (configuraciones de Vercel y Railway), más documentación técnica y de usuario revisada.
Resultados de pruebas, evidencias y documentos de gestión de cambios, configuración, riesgos y calidad elaborados en la materia.
Propósito: servir como referencia oficial del producto entregado, contra la cual se puedan evaluar futuras mejoras o mantenimientos, y como evidencia de cumplimiento de los objetivos de calidad y alcance.

Cada línea base se asocia a un conjunto concreto de versiones en el repositorio GitHub y a versiones específicas de documentos y esquemas de base de datos, de modo que cualquier modificación posterior debe pasar por un proceso explícito de control de cambios.

11. Resultados del Proyecto
 Funcionalidades implementadas
El proyecto Form-Creator logró implementar una plataforma web responsiva orientada a la creación, distribución, respuesta y análisis de formularios empresariales. De acuerdo con el alcance definido, el sistema cubre el ciclo de vida completo del formulario, desde su diseño hasta la visualización de resultados, integrando módulos de gestión de usuarios, gestión de formularios y gestión de respuestas.
En el módulo de gestión de usuarios, se implementaron funcionalidades tales:
Registro de nuevos usuarios
Inicio de sesión tradicional/Autenticación mediante Google/Firebase
Recuperación de contraseña por OTP 
Edición de la información del perfil. 

En el módulo de creación y gestión de formularios, el sistema permite al administrador:
 Diseñar formularios personalizados con preguntas de texto libre, opción múltiple y escala numérica.
Definir obligatoriedad y validaciones
Configurar visibilidad
Guardar borradores
Publicar y despublicar formularios, editarlos, duplicarlos y eliminarlos. 
También se habilitó la generación de enlaces únicos de acceso, lo que facilita la distribución controlada sin requerir autenticación obligatoria en formularios públicos. Todas las historias del backlog asociadas a estas capacidades están completadas y el soporte de ello se refleja tanto en el Jira como en el documento de lineamientos del proyecto. 
En el módulo de respuesta y análisis, se implementó 
El diligenciamiento público mediante enlace
La validación de campos obligatorios
El almacenamiento persistente de respuestas
La posibilidad de editar respuestas si el formulario lo permite
El envío de copia de respuestas por correo electrónico 
La visualización de estadísticas mediante gráficos. El cierre del Sprint 4 evidencia que la funcionalidad de visualización de datos con gráficos fue completada y que el sistema pudo mostrar el producto final ya desplegado y utilizable para los stakeholders.
Como resultado funcional global, Form-Creator dejó de ser únicamente un recolector de respuestas para convertirse en una solución integral de captura, administración y análisis de información, permitiendo que el usuario administrador gestione formularios y que el respondedor interactúe de manera simple, accesible y validada.

Desempeño del sistema
El desempeño del sistema fue planteado desde los requerimientos no funcionales con énfasis en disponibilidad, seguridad, compatibilidad y soporte a concurrencia. 
A nivel técnico, el sistema se construyó con una arquitectura desacoplada entre React en frontend y Django + Django REST Framework en backend, con MongoDB como base de datos. Esta decisión favoreció la mantenibilidad y una interacción fluida entre cliente y servidor mediante JSON y endpoints REST. Asimismo, el uso de Recharts permitió mostrar gráficos interactivos que se actualizan dinámicamente, mientras que React facilitó validaciones en tiempo real y respuesta inmediata a las acciones del usuario.
En cuanto a ejecución, el proyecto fue probado en desarrollo con React, Django y MongoDB Atlas, y desplegado en producción usando Vercel, Railway y MongoDB Atlas, lo cual permitió contar con una versión funcional accesible desde la web. El hecho de haber llegado a despliegue productivo y sustentación final demuestra un nivel de desempeño suficiente para cumplir el propósito académico y operativo del proyecto. No obstante, se reconoce como aprendizaje la necesidad de reforzar pruebas post despliegue y considerar limitaciones de servidores gratuitos, por lo que el rendimiento logrado puede considerarse satisfactorio, aunque todavía susceptible de optimización.
Concluyendo, el sistema respondió adecuadamente a operaciones clave como autenticación, creación de formularios, respuesta mediante enlace, validaciones, exportación y visualización gráfica. Esto evidencia que el software alcanzó un desempeño coherente con su alcance: ofrecer una experiencia accesible para el respondedor y una experiencia administrativa robusta para el creador del formulario.
Cumplimiento de objetivos
El objetivo general del proyecto consistía en desarrollar una plataforma web capaz de crear formularios personalizables, recopilar respuestas de forma centralizada y generar visualizaciones simples en tiempo real. A partir de las funcionalidades implementadas y del cierre de las historias del backlog, puede afirmarse que este objetivo fue cumplido de manera satisfactoria.
Añadiendo, el sistema cumplió con metas de negocio importantes definidas en el análisis, como optimizar el ciclo de recolección de datos, democratizar el análisis para usuarios no técnicos, eliminar barreras técnicas en la creación de formularios y mejorar la calidad de la información desde el origen a través de validaciones. 
Finalmente, estos resultados muestran que Form-Creator no solo cumplió con los objetivos, sino que también materializó una solución funcional y coherente con una necesidad real.

12. Conclusiones
El desarrollo de Form-Creator permitió comprobar que una necesidad frecuente en empresas y organizaciones —la recolección y análisis de datos mediante formularios— puede resolverse eficazmente mediante una plataforma web bien estructurada. La solución construida alcanzó un nivel funcional que permite crear, publicar, responder y analizar formularios, integrando autenticación, validaciones, persistencia de datos y reportes gráficos dentro de un mismo sistema. 
Uno de los principales aprendizajes obtenidos fue la importancia de definir correctamente el modelo de datos, documentar historias de usuario, criterios de aceptación y casos de uso, y mantener trazabilidad entre análisis, diseño e implementación. También se evidenció el valor de coordinar adecuadamente frontend y backend, especialmente cuando se manejan cambios en estructuras de datos, integraciones externas y validaciones cruzadas entre cliente y servidor. 
El proyecto también permitió reafirmar la importancia de aplicar ingeniería de software de manera rigurosa. La combinación de un enfoque híbrido entre documentación formal y prácticas ágiles fue adecuada para el contexto académico del equipo. La ingeniería de software aportó orden, trazabilidad, claridad de requisitos, control del alcance y capacidad de adaptación durante la implementación. Gracias a ello, el equipo pudo avanzar por sprints con una base documental sólida y con entregas incrementales que fueron refinando la solución sin perder coherencia con los objetivos iniciales.
En cuanto a las dificultades encontradas, el equipo de trabajo evidencia retos asociados a la integración frontend-backend, a la necesidad de mejorar la cobertura de pruebas, a la dependencia de servicios externos como correo transaccional y Firebase, y a las limitaciones del despliegue sobre infraestructura gratuita. También existieron desafíos relacionados con la consistencia de validaciones, la edición de formularios ya respondidos y la necesidad de preservar integridad en la base de datos. Sin embargo, estas dificultades no impidieron la entrega del sistema; por el contrario, se convirtieron en oportunidades de aprendizaje técnico y metodológico para el equipo.
En síntesis, Form-Creator demuestra que la aplicación disciplinada de la ingeniería de software permite transformar una idea académica en una solución real, funcional y extensible. Aunque el sistema todavía tiene oportunidades de mejora, el proyecto logró establecer una base sólida tanto desde el punto de vista tecnológico como desde la perspectiva del proceso de desarrollo.
13. Recomendaciones
Mejora del sistema
Se recomienda fortalecer la plataforma en aspectos de robustez, escalabilidad y experiencia de usuario. 
Desde el punto de vista técnico, sería recomendable robustecer la seguridad con mejores mecanismos de encriptación de contraseña y controles de sesión. 
En cuanto a la experiencia de usuario, se piensa en formas más accesibles para compartir y acceder a encuestas, a través de diferentes medios y en otros formatos como QR. Actualmente se utilizan enlaces compartidos por correo electrónico. 
Por otro lado, una funcionalidad adicional útil para el sistema también es la exportación de gráficas en pdf. Esta feature está prevista por experiencia de usuario, con el fin de que este tenga una forma de acceder a su información rápidamente sin obligarlo a ingresar a la web. Esto se alinea con el objetivo general del sistema. 
Mejoras en la gestión del proyecto
Convendría mantener y profundizar el enfoque híbrido utilizado, pero reforzando la etapa de pruebas y control de calidad dentro de cada sprint. Las retrospectivas muestran que se logró una buena organización mediante Scrum, aunque sería beneficioso institucionalizar mejor las pruebas técnicas antes de cada review y después del despliegue.
Asimismo, se requiere  mejorar la planificación del backlog con una mayor precisión en la dependencia entre historias y en el impacto de cambios estructurales, sobre todo cuando se afectan formularios publicados o respuestas persistidas. También sería útil mantener una documentación viva más compacta y orientada a cambios incrementales, para que el mantenimiento evolutivo sea más simple y menos costoso en tiempo.
Trabajo futuro
Form-Creator podría incorporar varias de las funcionalidades que quedaron expresamente fuera del alcance actual: autenticación avanzada, lógica condicional compleja en formularios, carga de archivos, reportes más sofisticados. Estas extensiones serían coherentes con la arquitectura modular ya definida.
También sería pertinente ampliar el módulo analítico con exportación a otros formatos, visualizaciones comparativas, análisis por periodos, indicadores de desempeño y paneles más cercanos a inteligencia de negocio.
